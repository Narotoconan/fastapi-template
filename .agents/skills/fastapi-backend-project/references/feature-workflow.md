# 跨层功能工作流

在新增或重塑正式业务功能、修改公共 API、接入缓存或调整中间件时读取本文件。

## 建模变更

1. 明确请求、响应、权限、业务状态、失败语义和兼容要求。
2. 追踪现有调用链、装配入口、测试与 `SKILL.md` 中对应的权威文档。
3. 标出业务不变量由哪一层执行，以及最终由数据库约束、权限依赖或外部系统中的哪一项保证。
4. 确定事务开始与结束位置，并区分事务内操作、提交后操作和允许降级的操作。
5. 列出兼容策略、迁移或回填要求、缓存失效范围和可观察的失败结果。

## 影响矩阵

| 变化 | 必查事项 |
| --- | --- |
| 公共 API | 调用方、OpenAPI、成功与错误响应、兼容窗口、对应文档 |
| 持久化结构 | 数据约束、索引、迁移、历史数据、回滚与部署顺序 |
| 缓存 | 键空间、序列化、TTL、提交后失效、失败策略与补偿 |
| 中间件或生命周期 | 注册顺序、启动失败、关闭清理、协议头与异常响应 |
| 外部服务 | 超时、重试、幂等、降级、敏感信息与集成测试 |

## 实施顺序

1. 先锁定公共契约与兼容策略，避免数据库或实现细节反向决定 API。
2. 设计迁移、历史数据处理、回滚和部署顺序，再落持久化结构。
3. 标出事务内的原子操作、数据库约束和预期失败转换。
4. 单独设计提交后的缓存失效、外部副作用、失败策略和补偿。
5. 最后完成装配与针对性验证，确保影响矩阵中的每个边界都有证据。

## 交付证据

- 明确事务边界以及哪些动作发生在成功提交之后。
- 明确迁移、部署顺序、回滚和历史数据假设。
- 明确缓存与外部依赖的降级、一致性和补偿选择。
- 列出已验证行为，以及仍需真实 PostgreSQL、Redis 或外部服务验证的部分。
